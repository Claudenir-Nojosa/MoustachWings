Até agora, estilizar um artigo, documento ou postagem de blog com o Tailwind tem sido uma tarefa tediosa que exigia um olhar atento para a tipografia e uma grande quantidade de CSS personalizado complexo.

Por padrão, o Tailwind remove todo o estilo padrão do navegador de parágrafos, cabeçalhos, listas e muito mais. Isso acaba sendo muito útil para construir interfaces de aplicativos, porque você gasta menos tempo desfazendo os estilos do usuário, mas quando você _realmente_ está tentando estilizar algum conteúdo que veio de um editor de texto rico em um CMS ou de um arquivo Markdown, pode ser surpreendente e não intuitivo.

Recebemos muitas reclamações sobre isso, com as pessoas nos perguntando regularmente coisas como:

> Por que o Tailwind está removendo os estilos padrão dos meus elementos `h1`? Como desabilito isso? O que você quer dizer com perder todos os outros estilos básicos?
> Nós entendemos você, mas não estamos convencidos de que simplesmente desabilitar nossos estilos básicos é o que você realmente quer. Você não quer ter que remover margens irritantes toda vez que usar um elemento `p` em um painel da sua interface. E duvido que você realmente queira que suas postagens de blog usem os estilos do usuário também - você quer que elas pareçam _incríveis_, não terríveis.

O plugin `@tailwindcss/typography` é a nossa tentativa de oferecer o que você _realmente_ deseja, sem nenhum dos inconvenientes de desabilitar nossos estilos básicos.

Ele adiciona uma nova classe `prosa` que você pode aplicar a qualquer bloco de conteúdo HTML simples e transformá-lo em um documento bonito e bem formatado:

```ts
import * as React from "react"
import { AlternarModo } from "@/componentes/alternar-modo"
import { Ícones } from "@/componentes/ícones"

import { configSite } from "@/config/site"
import { cn } from "@/lib/utils"

export function RodapéDoSite({ className }: React.HTMLAttributes<HTMLElement>) {
  return (
    <rodapé className={cn(className)}>
      <div className="contêiner flex flex-col items-center justify-between gap-4 py-10 md:h-24 md:flex-row md:py-0">
        <div className="flex flex-col items-center gap-4 px-8 md:flex-row md:gap-2 md:px-0">
          <Ícones.logo />
          <p className="text-center text-sm leading-loose md:text-left">
            <a
              href={configSite.links.twitter}
              target="_blank"
              rel="noreferrer"
              className="font-medium sublinhado offset-sublinhado-4"
            >
              Claudenir Nojosa
            </a>
            .
          </p>
        </div>
      </div>
    </rodapé>
  )
}
```

Para obter mais informações sobre como usar o plugin e os recursos que ele inclui, [leia a documentação](https://github.com/tailwindcss/typography/blob/master/README.md).

## O que esperar daqui para frente

O que se segue a partir daqui é apenas um monte de bobagens absolutas que escrevi para testar o próprio plugin. Ele inclui todos os elementos tipográficos sensatos que consegui pensar, como **texto em negrito**, listas não ordenadas, listas ordenadas, blocos de código, citações em bloco e até _itálicos_.

É importante cobrir todos esses casos de uso por algumas razões:

1.  Queremos que tudo pareça bom "pronto para uso".
2.  Realmente, apenas a primeira razão, esse é o ponto principal do plugin.
3.  Aqui está uma terceira razão fictícia, embora uma lista com três itens pareça mais realista do que uma lista com dois itens.

Agora vamos experimentar outro estilo de cabeçalho.

## E se empilharmos cabeçalhos?

### Devemos garantir que isso também pareça bom.

Às vezes, você tem cabeçalhos diretamente um abaixo do outro. Nestes casos, você frequentemente precisa desfazer a margem superior no segundo cabeçalho porque geralmente fica melhor que os cabeçalhos fiquem mais próximos do que um parágrafo seguido de um cabeçalho deve ser.

### Quando um cabeçalho vem após um parágrafo ...

Quando um cabeçalho vem após um parágrafo, precisamos de um pouco mais de espaço, como mencionei acima. Agora vamos ver como uma lista mais complexa ficaria.

- **Eu costumo fazer isso em que os itens da lista têm cabeçalhos.**

  Por algum motivo, acho que isso parece legal, o que é lamentável, já que é bem irritante acertar os estilos.

  Eu costumo ter dois ou três parágrafos nesses itens de lista, então a parte difícil é acertar o espaçamento entre os parágrafos, o cabeçalho do item da lista e os itens da lista separados para que tudo faça sentido. Bem complicado, honestamente, você poderia argumentar com convicção que você simplesmente não deveria escrever assim.

- **Já que se trata de uma lista, eu preciso de pelo menos dois itens.**

  Já expliquei o que estou fazendo no item de lista anterior, mas uma lista não seria uma lista se tivesse apenas um item, e realmente queremos que isso pareça realista. É por isso que adicionei este segundo item de lista para que eu tenha algo para olhar ao escrever os estilos.

- **Não é uma má ideia adicionar um terceiro item também.**

  Acho que provavelmente teria sido suficiente usar dois itens, mas três definitivamente não é pior, e como parece que não tenho dificuldade em inventar coisas arbitrárias para digitar, posso muito bem incluí-lo.

Depois desse tipo de lista, costumo ter uma declaração de encerramento ou parágrafo, porque meio que parece estranho pular direto para um cabeçalho.

## O código deve parecer bom por padrão.

Acredito que a maioria das pessoas vai usar [highlight.js](https://highlightjs.org/) ou [Prism](https://prismjs.com/) ou algo do tipo se quiser estilizar seus blocos de código, mas não custa nada fazer com que eles pareçam _bons_ desde o início, mesmo sem destaque de sintaxe.

Aqui está como um arquivo `tailwind.config.js` padrão parece no momento da escrita:

```js
module.exports = {
  purge: [],
  theme: {
    extend: {},
  },
  variants: {},
  plugins: [],
}
```
